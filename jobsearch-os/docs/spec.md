## A14. Interview Story Bank (STAR)

### Story 1 — DTD4X CRM Follow-Up System
- **Situation:** Managing a ~400-lead real estate pipeline where follow-up was inconsistent and leads were going cold — classic pipeline leakage.
- **Task:** Create a follow-up process that was repeatable, measurable, and simple enough to sustain at volume.
- **Action:** Designed DTD4X — 4 full database touches per quarter (16/year) across defined channels: call, text, email/social, and a value-add touch. Built it into Follow Up Boss with stages, tags, and Smart Lists; tracked cadence completion and response rate by channel; used channel data to adjust the mix.
- **Result:** Inconsistent, memory-based follow-up became a scheduled operating process with completion tracking. Cold-lead reactivation and touch consistency became measurable instead of anecdotal.

# PART B — JOB-SEARCH OPERATING SYSTEM

## B1. Executive Summary of the System

A daily-use job-search operating system built around one idea: **diagnose the company's problem before touching the application.** Every job that enters the system gets (1) a business-problem diagnosis, (2) a 1–100 fit score against my honest skillset, (3) a recommendation (Strong Apply / Tailored Apply / Research / Skip), and — for pursued roles — (4) a tailored cover letter, (5) a ranked contact list, and (6) a Steve Dalton-style 4-touch outreach sequence with every message drafted into an **approval queue**. Nothing sends automatically. I open the dashboard each day, see exactly what's due, approve or edit drafts, execute, and log outcomes. Weekly metrics show which titles, industries, and messages actually convert, so the system self-corrects.

Build target: Python + SQLite backend, CSV import/export, and a single-page local HTML dashboard. One hour per day of my time; the system does the reading, drafting, scoring, and sequencing.

---

## B2. The Exact Problem This System Solves

- **Random applying:** I was spraying applications without knowing why each company opened the role. This system forces a diagnosis first and kills weak-fit roles before I waste time on them.
- **One-touch outreach:** Messages went out once, then died. This system enforces a tracked 4-touch sequence with due dates.
- **No memory:** Who did I message? When? What did I say? Everything is now a record with a status and a next action.
- **Positioning drift:** Under deadline pressure, cover letters drift into buzzwords or inflated claims. Every generated artifact is constrained by Part A's language rules.
- **No feedback loop:** I couldn't see which titles or messages performed. Weekly metrics answer that.
- **Motivation structure:** I've established that I execute best with external structure. The daily dashboard *is* that structure — a queue that tells me exactly what today requires.

## B3. Job/Company Diagnosis Framework

For every job posting ingested, Codex runs this analysis on the full JD text and produces a **Diagnosis Record**. It must infer, not summarize.

**The 10 diagnosis questions (stored per job):**
1. What is the company likely struggling with right now? (Read between the lines: "improve data hygiene" = the CRM is a mess; "build reporting from scratch" = leadership is flying blind; "support a growing sales team" = process broke at scale; backfill language = someone left and tribal knowledge left with them.)
2. What operational problem is this role supposed to solve?
3. What systems, data, processes, teams, or workflows are likely messy? (Name the probable tools from JD clues.)
4. What does success look like at 90 days?
5. What does success look like at 6 months?
6. Which parts of my background directly match that problem? (Pull from A2/A3 only.)
7. Which proof points should be used? (Max 3, ranked; pull from A3.)
8. What language must be avoided for this role? (Pull from A11 + role-specific traps, e.g., don't imply Salesforce admin skills if JD wants an admin.)
9. What is my honest one-paragraph value proposition for this exact role?
10. Recommendation: **Apply / Tailored Apply / Research More / Skip** — with one sentence of reasoning.

**Diagnosis signal cheat sheet (for the inference step):**

| JD clue | Likely underlying problem |
|---|---|
| "Data hygiene," "clean up," "audit records" | Dirty CRM, unreliable reporting, leadership distrust of numbers |
| "Build dashboards/reporting" | No visibility; decisions made on gut feel |
| "Document processes," "create SOPs" | Tribal knowledge risk; recent turnover or scaling |
| "Cross-functional," "work with sales and marketing" | Handoff leakage between teams |
| "Fast-paced," "wear many hats" | Understaffed ops; first or second ops hire |
| "Support [tool] migration/implementation" | Mid-migration pain; need hands not strategy |
| "Improve lead routing/follow-up" | Pipeline leakage — my strongest match; flag these |
| "First RevOps hire" | Founder-led chaos; huge opportunity but scope risk |

**Auto-flags (set during diagnosis):**
- `PIPELINE_LEAKAGE` flag → boosts score; this is the DTD4X sweet spot.
- `RCM_HEALTHCARE` flag → "Revenue Cycle Analyst" postings that are healthcare billing/claims → auto-Skip with reason "domain mismatch (RCM ≠ RevOps)."
- `ADMIN_CERT_REQUIRED` flag → Salesforce Admin cert required → downgrade one tier; never claim admin skills.
- `SENIOR_SCOPE` flag → 5+ years required or "lead/manage team" → downgrade; apply only if score ≥85 on everything else.

## B6. Steve Dalton-Style Follow-Up Sequence

Four touches per contact, tracked with due dates. Tone: short, human, specific, zero desperation. All templates inherit from A9 and the messaging bank in A13.

**Touch 2 — Day 3–4 (after acceptance, or as InMail/message if open profile): role-specific reason.**
> "Thanks for connecting, [Name]. One reason I'm serious about this role: the posting reads like [diagnosed problem — e.g., 'follow-up consistency is the gap'], and that's exactly what I built DTD4X to fix — a quarterly cadence system with completion and response tracking. **Touch 3 — Day 8–10: email fallback if no LinkedIn response.**
Sender: **armaninevarez9@gmail.com**. Subject: `[Role] application — quick note from a candidate`.
> "Hi [Name], I applied for [Role] on [date] and reached out on LinkedIn — following up here in case that's easier. The short version: [diagnosed problem] is the kind of problem I've actually worked: [best proof point, one sentence, with metric]. **Sequence rules:**
- Max 2 active sequences per company (don't carpet-bomb one org).
- A response at any touch pauses the sequence and creates a `RESPOND` task instead.
- A "not interested / no" ends the sequence permanently and logs the contact as closed-polite.
- Recruiters get a compressed 3-touch version (they move faster or not at all).
- Every touch is drafted by the system, **sent by me** after approval.

## B13. Risks, Blind Spots, and Human-Approval Requirements

**Must never happen without explicit approval (hard gates):**
- Sending any message, connection request, or email
- Submitting any application
- Contacting anyone at my current/recent companies or anyone in my personal network
- Using a new message template at scale before it's been human-reviewed
- Any claim not traceable to Part A

**Risks and blind spots to manage:**
1. **The $22→$32/hour claim.** This document introduces "improved gross hourly earnings from ~$22 to ~$32/hour." The *validated* number in Part A is the $32.96 baseline over 59.1 hours. Before the improvement claim goes into any letter or interview, confirm the $22 starting figure is backed by actual Gridwise/Uber data from the early period. If it is, it becomes proof point #11 (before/after is stronger than a baseline). If it isn't cleanly documented, use the validated baseline only. **Default: baseline only, until confirmed.**
2. **Diagnosis overconfidence.** The system *infers* company problems; it will sometimes be wrong. Cover letters should frame diagnoses as informed reads ("postings like this usually mean…"), never as certainties about their internals.
3. **Volume creep.** The Dalton method fails if it becomes spray-and-pray with better formatting. Cap: ~5 new applications/day, ~10 new contacts/day. Depth beats volume.
4. **LinkedIn account risk.** No automation touches LinkedIn — ever. Manual lookups and manual sends only. Connection requests capped at ~15/week to stay under spam-detection patterns.
5. **Template staleness.** If the same T1 goes to many people at overlapping companies, it gets recognized. The uniqueness check on cover letter openings should extend to T1 messages within the same company.
6. **Timeline questions.** Overlapping employment dates (Surge/ComEd concurrent with JLL and KW) will come up in screens. Prepared one-liner required before first phone screen — this is a me-task, not a system task.
7. **Metrics gaming.** Don't optimize for "touches sent." The only metrics that matter are responses, calls, interviews. Dashboard ranks outcome metrics above activity metrics deliberately.
8. **Email deliverability.** A Gmail address sending similar-looking messages can get flagged. Keep T3 emails genuinely varied, low-volume (<10/day), no links in first contact beyond LinkedIn/GitHub.
